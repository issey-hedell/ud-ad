# Claude Code 運用ルール

**目的**: チームで Claude Code を使う際の手順を統一し、再現性を確保する

---

## 1. 会話開始時のルール（必須）

### 新しい会話を始める時

**必ず以下のテンプレートで開始する**

```
まず CLAUDE.md を読んでから作業してください。

【依頼内容】
〇〇をやりたい
```

### なぜ必要か

- Claude Code は新しい会話を始めると、前の文脈を忘れる
- CLAUDE.md を読ませないと、プロジェクトのルールを無視して作業する
- 毎回同じ指示を入れることで、誰がやっても同じ品質になる

---

## 2. CLAUDE.md の配置ルール

### 配置場所

```
プロジェクト名/
├── CLAUDE.md          ← ルート直下に必ず配置
├── src/
├── ...
```

### CLAUDE.md に書く内容

```markdown
# プロジェクト名

## このプロジェクトについて
（プロジェクトの概要）

## 技術スタック
- 言語: 
- フレームワーク: 
- DB: 

## ディレクトリ構成
（主要なフォルダの説明）

## コーディングルール
（守ってほしいルール）

## 参照すべきドキュメント
- 設計書: docs/design.md
- UIパターンルール: docs/ui_pattern_rules.md
```

---

## 3. 作業依頼のルール

### 良い依頼の書き方

```
まず CLAUDE.md を読んでから作業してください。

【依頼内容】
ユーザー一覧画面を作成したい

【参考】
- 設計書: docs/design.md の「ユーザー一覧」セクション
- 参考UI: （スクショ or URL）

【注意点】
- 削除ボタンも必要
- ローディング表示を入れる
```

### 悪い依頼の書き方

```
ユーザー一覧画面作って
```

→ CLAUDE.md を読まない、参考情報がない、抜け漏れが発生する

---

## 4. 途中で文脈が切れた時

### 兆候

- 急にルールを無視し始める
- 前に話したことを忘れている
- おかしな実装をし始める

### 対処法

```
CLAUDE.md を再度読んでください。
その上で、〇〇を続けてください。
```

---

## 5. 長い作業を依頼する時

### 分割して依頼する

```
【悪い例】
「ユーザー管理機能を全部作って」

【良い例】
「まず、ユーザー一覧画面を作ってください」
→ 完了確認
「次に、ユーザー作成画面を作ってください」
→ 完了確認
```

### 理由

- 一度に大きな依頼をすると、途中で品質が落ちる
- 分割して確認することで、問題を早期発見できる

---

## 6. エラーが出た時の報告テンプレート

Claude Code にエラーを報告する時のフォーマット。

```
【画面】どの画面で
【操作】何をしたら
【結果】どうなった
【期待】本来どうなるべき
【エラーメッセージ】（あれば貼る）
【スクショ】（あれば貼る）
```

→ これをそのまま送れば、Claude Code が原因調査・修正できる

---

## 7. コミット前の確認ルール

Claude Code に実装させた後、コミット前に確認すること。

```
【確認項目】
- [ ] 意図した機能が実装されているか
- [ ] 余計なファイルが変更されていないか
- [ ] コンソールにエラーが出ていないか
```

---

## チェックリスト

### 会話開始時

- [ ] 「まず CLAUDE.md を読んでから」と指示した
- [ ] 依頼内容を明確に書いた
- [ ] 参考情報（設計書、参考UI）を添えた

### 作業中

- [ ] 大きな作業は分割して依頼した
- [ ] 各ステップで完了確認した

### 作業完了時

- [ ] 意図通りの実装か確認した
- [ ] コンソールエラーがないか確認した

---

## まとめ

```
【鉄則】
1. 新しい会話は必ず「CLAUDE.md を読んで」から始める
2. 依頼は具体的に、参考情報を添えて
3. 大きな作業は分割して依頼
4. 文脈が切れたら CLAUDE.md を再読み込み
```
